## Decisions honoured (from the Phase 0 checkpoint's D1-D7)

- **D1 (no `cs-*`)**: `spacing.*` and `radius.*` follow the Tailwind default scale. No overlap.
- **D2 (single red)**: only `color.semantic.error.fg`. `laser-red` does not exist in Pharos.
- **D3 (body 14px)**: `font.size.sm` = 14px/0.875rem. `$description` documents that it is the body default.
- **D4 (Lucide)**: applies to pharos-react, not pharos-tokens. No impact here.
- **D5 (Alexandria's root `components/`)**: applies to Phase 6. No impact here.
- **D6 (no legacy aliases)**: zero deprecated tokens. `laser-red`, `muted-black`, `charcoal-blue`, etc., are NOT in Pharos.
- **D7 (Phase 0 reports)**: the location rule in alexandria-web-application is respected. This `REPORT-fase-1A.md` lives inside pharos-tokens/ for consistency with the repo's scope.

## Remaining work

### Blockers before publishing 0.1.0

1. **`NPM_TOKEN` in GitHub secrets** of the repo `code-sherpas/pharos-tokens`. Needs `publish` permission on the `@code-sherpas` scope. When you merge this PR to `main`, the release workflow opens a "Version packages" PR; when that one is merged, it publishes automatically to npm. Manual alternative: `pnpm changeset version && pnpm release` with a local NPM token.
2. **Branch protection** on `main` of pharos-tokens: require CI green. Configured in Settings ‚Üí Branches on GitHub (not code, cannot be automated here without gh CLI auth).

### Optional follow-up (Phase 1B+ or 0.1.x patches)

1. **Real font family**: today `font.family.sans` uses `"Inter", system-ui, ...` as a generic fallback. If Alexandria loads a specific font via `next/font` (I did not verify `RootLayout.tsx` in Phase 0), Pharos should mirror it as an official token.
2. **Accent colors**: no `color.accent.*` added for bluish-purple (`#6f42f5`) or indochine (`#c27605`) because they do not have broad use in Alexandria. If the designer (CTO) considers them part of the system, they are added as a minor bump.
3. **"on" contrast tokens for neutrals**: there is no `neutral.{X}.on` right now. In Phase 2 when components (especially `Button variant="secondary"`) need them, add. For now consumers use `color.base.white` or `color.neutral.900` manually.
4. **Richer animation tokens**: only basic `duration` + `easing`. If the need for animation `delay`, `iteration-count`, etc., appears, add a category.
